# Cloud/MSA 실습 과제 보고서

## 1. 개요

본 실습의 목표는 기존 개발 환경을 기반으로 운영 환경의 가용성과 확장성을 높이기 위한 아키텍처를 설계하고, 이를 퍼블릭 클라우드와 컨테이너 기반의 마이크로서비스 구조 위에 구현하는 것이다. 본 보고서는 평가 기준에 맞추어 Value - Architect - Build - Measure - Evaluation 순서로 정리하였다.

상단 제출 정보는 아래와 같다.

| 항목 | 내용 |
| --- | --- |
| handson-cloud-msa-outer 저장소 URL | 입력 필요 |
| eshop-for-ci-cd 저장소 URL | 입력 필요 |
| 제출자 | 입력 필요 |
| 제출 일시 | 입력 필요 |

## 2. Value

### 2.1 품질 목표

본 실습에서는 단순히 "서비스가 죽지 않아야 한다" 수준이 아니라, 운영 환경에서 실제로 검증 가능한 가용성 및 확장성 시나리오를 정의하였다. 핵심 목표는 다음 4가지이다.

1. 트래픽 증가로 인해 단일 인스턴스 장애가 전체 장애로 확산되지 않아야 한다.
2. 코드 변경 배포 시 사용자 서비스 중단 없이 신규 버전으로 전환되어야 한다.
3. 사용자 요청이 몰리는 기능만 선택적으로 확장할 수 있어야 한다.
4. 핵심 기능과 비핵심 기능을 분리하여 비핵심 기능 장애 시에도 주문 흐름이 유지되어야 한다.

### 2.2 품질 속성 시나리오

#### 시나리오 V-1. 무중단 배포

- 자극: 운영 중인 서비스에 신규 버전이 배포된다.
- 환경: GKE 기반 운영 클러스터, 다중 Pod 구성
- 대상: `frontend`, `backend`, `cartservice`, `productservice`
- 반응: 기존 Pod가 모두 제거되기 전에 신규 Pod가 순차적으로 반영되어 서비스 요청이 계속 처리된다.
- 측정 기준: 배포 중 외부 사용자 요청 성공률 99% 이상, HTTP 5xx 비율 1% 미만

#### 시나리오 V-2. 애플리케이션 인스턴스 장애 허용

- 자극: Web/WAS 계층의 일부 Pod가 강제 종료된다.
- 환경: 핵심 서비스 최소 2개 이상 인스턴스 운영
- 대상: `frontend`, `backend`, `cartservice`, `productservice`
- 반응: 남은 인스턴스로 트래픽이 우회되어 상품 조회, 장바구니, 주문 기능이 지속된다.
- 측정 기준: 단일 Pod 종료 시 핵심 기능 요청 성공률 95% 이상

#### 시나리오 V-3. 데이터 계층 장애 허용

- 자극: Redis, PostgreSQL, MongoDB, RabbitMQ 인스턴스 일부가 중단된다.
- 환경: HA 구성 및 failover 지원 구조
- 대상: 각 마이크로서비스가 사용하는 전용 데이터 계층
- 반응: 다른 노드 또는 리더로 failover 되어 전체 서비스 중단 없이 핵심 기능이 유지된다.
- 측정 기준: 단일 DB/메시지 브로커 인스턴스 장애 시 전체 서비스 중단 없음

#### 시나리오 V-4. 선택적 자동 확장

- 자극: 특정 서비스 CPU 사용률이 최대 부하의 50%에 도달한다.
- 환경: HPA 적용 대상 서비스와 GKE Cluster Autoscaling 사용
- 대상: `frontend`, `backend`, `cartservice`, `productservice`
- 반응: 해당 서비스 Pod 수가 자동 증가하고, 필요 시 Node Pool이 자동 확장된다.
- 측정 기준: CPU 50% 도달 후 HPA가 scale-out 수행, 저부하 상태 지속 시 점진적 scale-in 수행

#### 시나리오 V-5. 비핵심 기능 장애 격리

- 자극: 추천 또는 광고 서비스가 중단된다.
- 환경: 핵심 기능과 비핵심 기능이 별도 서비스로 분리된 마이크로서비스 구조
- 대상: `recommendservice`, `adservice`
- 반응: 상품 조회, 장바구니, 주문 등 핵심 사용자 여정은 정상 동작한다.
- 측정 기준: 비핵심 서비스 중단 시 핵심 기능 API 성공률 95% 이상

## 3. Architect

### 3.1 설계 개요

운영 환경은 GCP 상의 GKE 기반 Kubernetes 클러스터로 구성하였다. 인프라는 Terraform 코드로 관리하고, 운영 애플리케이션은 Kubernetes 선언형 매니페스트로 관리하여 GitOps 방식에 적합한 구조로 설계하였다. 관리 영역에서는 Jenkins와 Argo CD를 유지하고, 운영 영역에서는 핵심 기능 서비스에 대해 다중 인스턴스와 자동 확장을 적용하였다.

### 3.2 주요 설계 결정과 대안 분석

#### 결정 A. 운영 플랫폼 선택

대안 1. VM 기반 Web/WAS 수동 이중화

- 장점: 기존 구조와 유사하여 이해가 쉽다.
- 단점: 확장과 복구가 수동 중심이며, 서비스 단위 독립 확장이 어렵다.

대안 2. Kubernetes 기반 마이크로서비스 운영

- 장점: 서비스 단위 배포, HPA, self-healing, rolling update, node autoscaling 활용 가능
- 단점: 학습 비용과 운영 복잡성이 증가한다.

선택: 대안 2

선택 이유는 본 과제의 핵심 요구사항이 가용성과 확장성 확보이며, Kubernetes는 이를 위한 표준 기능을 기본 제공하기 때문이다. 특히 일부 기능만 선택적으로 scale-out 해야 하는 조건을 만족시키기 위해서는 서비스별 독립 배포와 오토스케일링이 가능한 구조가 더 적합하다.

#### 결정 B. 데이터 계층 구성 방식

대안 1. 모든 서비스가 하나의 공용 DB를 공유

- 장점: 초기 구축이 단순하다.
- 단점: 장애 전파 범위가 넓고 서비스 독립성이 떨어진다.

대안 2. 서비스별 데이터 저장소를 분리하고 각 저장소에 HA를 적용

- 장점: 장애 격리, 데이터 소유권 분리, 서비스 단위 확장에 유리
- 단점: 운영해야 할 데이터 구성요소 수가 증가한다.

선택: 대안 2

선택 이유는 과제 요구사항에서 서비스별 별도 DB 확보와 장애 시 다른 서비스 영향 최소화를 강조하고 있기 때문이다. 본 구현에서는 PostgreSQL, MongoDB, Redis, RabbitMQ를 서비스 특성에 맞게 분리하여 사용하였다.

#### 결정 C. 비핵심 기능 처리 방식

대안 1. 추천/광고 기능을 핵심 백엔드에 포함

- 장점: 호출 구조가 단순하다.
- 단점: 비핵심 기능 장애가 핵심 주문 흐름에 전파될 수 있다.

대안 2. 추천/광고 기능을 별도 서비스로 분리

- 장점: 장애 격리와 독립 배포가 가능하다.
- 단점: 네트워크 호출 및 운영 대상이 증가한다.

선택: 대안 2

선택 이유는 비핵심 기능을 독립 서비스로 분리해야 장애가 핵심 흐름에 전파되지 않기 때문이다. `recommendservice`, `adservice`는 라우팅과 배포 단위가 분리되어 있으며, 핵심 기능과 구조적으로 독립적이다.

#### 결정 D. 배포 방식 선택

대안 1. 운영 서버에 직접 배포

- 장점: 초기 설정이 단순하다.
- 단점: 변경 이력 추적, 재현성, 승인 흐름 관리가 어렵다.

대안 2. Jenkins + Argo CD 기반 CI/CD + GitOps

- 장점: 선언형 배포, 변경 이력 추적, 환경 재현성 확보
- 단점: 관리 컴포넌트 구성이 추가로 필요하다.

선택: 대안 2

선택 이유는 과제 제약사항에서 Jenkins와 Argo CD를 그대로 사용해야 하며, 운영 환경이 Git 저장소에 코드 형태로 저장되어야 하기 때문이다.

### 3.3 아키텍처 View 1: 배치/인프라 View

1. 관리 클러스터
- Terraform으로 Jenkins와 Argo CD를 설치한다.
- Gateway와 DNS를 통해 외부에서 Jenkins, Argo CD에 접근한다.

2. 운영 클러스터
- GKE Node Pool은 `n1-standard-4` 머신 타입으로 구성한다.
- Node Pool은 최소 2대, 최대 4대 자동 확장으로 설정한다.
- 운영 네임스페이스 `eshop`에 마이크로서비스와 데이터 계층을 배포한다.

3. 외부 진입점
- GKE Gateway와 HTTPRoute로 `frontend`, `backend`, 각 API 서비스로 트래픽을 분기한다.

### 3.4 아키텍처 View 2: 논리/서비스 View

1. 사용자 요청은 `frontend` 또는 `/api` 경로를 통해 `backend`와 개별 서비스로 전달된다.
2. 핵심 서비스는 `frontend`, `backend`, `cartservice`, `productservice`로 정의하였다.
3. 비핵심 서비스는 `recommendservice`, `adservice`로 정의하였다.
4. 데이터 계층은 서비스 특성별로 분리하였다.
- 주문 및 핵심 비즈니스 데이터: PostgreSQL HA
- 상품 데이터: MongoDB ReplicaSet
- 장바구니 캐시: Redis Replication + Sentinel
- 비동기 메시징: RabbitMQ Cluster

### 3.5 설계 근거가 되는 코드 반영 사항

- 운영 GKE Node Pool은 `initial_node_count = 2`, `min_node_count = 2`, `max_node_count = 4`, `machine_type = "n1-standard-4"`로 정의되어 있다.
- 핵심 서비스 `frontend`, `backend`, `cartservice`, `productservice`에는 HPA가 적용되어 있으며 CPU 50%를 기준으로 scale-out 하도록 구성되어 있다.
- `recommendservice`, `adservice`, `currencyservice`는 독립 서비스로 분리되어 장애 격리가 가능하다.
- PostgreSQL은 CloudNativePG `instances: 3`으로 구성되어 있다.
- MongoDB는 ReplicaSet `members: 3`으로 구성되어 있다.
- Redis는 Replication `clusterSize: 3`과 Sentinel `clusterSize: 3`, `quorum: 2`로 구성되어 있다.
- RabbitMQ는 `replicas: 3` 클러스터로 구성되어 있다.

## 4. Build

### 4.1 구현 결과 요약

본 과제에서는 구현 과정 자체보다 최종 구현 상태를 중심으로 정리한다.

1. 인프라 코드화
- `environments/prod` 하위에 운영 환경 Terraform 코드가 정의되어 있다.
- 네트워크, GKE, 인증서, 출력값을 코드로 관리한다.

2. 운영 클러스터 구성
- GKE 운영 클러스터와 별도 Node Pool을 생성한다.
- 운영 Node Pool은 최소 2대 이상으로 구성하여 단일 노드 장애를 허용한다.

3. 애플리케이션 배포 구성
- `environments/prod/k8s` 하위에 서비스별 Kubernetes 매니페스트를 선언형으로 관리한다.
- `frontend`, `backend`, `cartservice`, `productservice`는 HPA 기반 자동 확장을 적용하였다.
- `gateway.yaml`과 `health-check-policy.yaml`로 외부 라우팅과 헬스체크를 구성하였다.

4. 데이터 계층 고가용성 구성
- PostgreSQL은 CloudNativePG 기반 다중 인스턴스 클러스터로 구성하였다.
- MongoDB는 ReplicaSet으로 구성하였다.
- Redis는 Replication + Sentinel 구조로 구성하였다.
- RabbitMQ는 다중 replica 클러스터로 구성하였다.

5. 관리 환경 구성
- `management` 하위 Terraform 코드로 Jenkins, Argo CD, Gateway를 구성하였다.
- Jenkins는 Terraform 실행 및 배포 자동화를 담당한다.
- Argo CD는 Git 저장소의 선언형 리소스를 기준으로 운영 상태를 동기화하는 역할을 담당한다.

### 4.2 운영 아키텍처 특징

본 아키텍처의 핵심은 "모든 것을 크게 확장하는 방식"이 아니라 "부하가 집중되는 서비스만 확장하는 방식"이다. 이를 위해 상품 조회, 장바구니, 주문 흐름에 직접 영향을 주는 서비스는 독립 배포 단위이자 독립 확장 단위로 설계하였다. 또한 데이터 계층도 서비스 성격에 따라 분리하여 한 데이터 저장소의 장애가 전체 서비스 장애로 확산되지 않도록 하였다.

## 5. Measure

### 5.1 검증 전략

Value 단계에서 정의한 시나리오를 기준으로 실제 운영 환경에서 다음 검증을 수행한다.

### 5.2 검증 항목별 결과

#### 검증 M-1. 무중단 배포 검증

검증 방법

1. 핵심 서비스 중 하나에 신규 이미지 태그를 반영한다.
2. Argo CD 또는 Jenkins 파이프라인을 통해 운영 반영을 수행한다.
3. 배포 중 반복 요청을 발생시켜 성공률과 오류율을 측정한다.

측정 예시

| 항목 | 측정값 |
| --- | --- |
| 대상 서비스 | 입력 필요 |
| 배포 시작 시각 | 입력 필요 |
| 총 요청 수 | 입력 필요 |
| 성공률 | 입력 필요 |
| 5xx 비율 | 입력 필요 |
| 결론 | 입력 필요 |

예상 해석

핵심 서비스는 다중 Pod 구조와 Kubernetes rolling update 특성에 의해 서비스 중단 없이 배포가 진행되어야 한다. 단, 현재 매니페스트에는 readinessProbe, PodDisruptionBudget이 명시되어 있지 않으므로 실제 결과가 기준에 미달할 경우 이 부분을 개선 포인트로 기록한다.

#### 검증 M-2. 애플리케이션 인스턴스 장애 검증

검증 방법

1. `frontend`, `backend`, `cartservice`, `productservice` 중 하나의 Pod를 강제 삭제한다.
2. 상품 조회, 장바구니 조회/추가, 주문 요청을 반복 수행한다.
3. 실패율과 복구 시간을 기록한다.

측정 예시

| 항목 | 측정값 |
| --- | --- |
| 장애 대상 Pod | 입력 필요 |
| 총 요청 수 | 입력 필요 |
| 성공률 | 입력 필요 |
| 평균 응답시간 | 입력 필요 |
| 복구 시간 | 입력 필요 |
| 결론 | 입력 필요 |

분석

핵심 서비스는 HPA 최소 replica 2 이상으로 설정되어 있으므로 단일 Pod 장애 시에도 서비스는 유지되어야 한다.

#### 검증 M-3. 데이터 계층 장애 검증

검증 방법

1. PostgreSQL, MongoDB, Redis, RabbitMQ 클러스터에서 단일 인스턴스를 중지하거나 Pod를 삭제한다.
2. 각 서비스 기능을 수행하며 장애 전파 여부를 확인한다.

#### 검증 M-4. 자동 확장 검증

검증 방법

1. `frontend`, `backend`, `cartservice`, `productservice` 중 하나에 부하를 집중시킨다.
2. CPU 사용률이 50%에 도달하도록 요청량을 증가시킨다.
3. HPA에 의해 replica 수가 증가하는지 확인한다.
4. 필요 시 Node 수가 증가하는지도 함께 확인한다.

## 6. Evaluation

### 6.1 Value 충족 여부 평가

본 아키텍처는 과제의 가용성 및 확장성 목표를 전반적으로 충족하는 방향으로 설계 및 구현되었다고 평가한다.

1. 가용성 측면
- 핵심 서비스에 대해 최소 2개 이상의 인스턴스를 유지하도록 설계하였다.
- GKE Node Pool도 최소 2대로 유지하여 단일 노드 장애를 허용한다.
- 데이터 계층은 단일 인스턴스가 아닌 HA 구성을 채택하여 장애 허용 범위를 높였다.
- 추천/광고 기능을 별도 서비스로 분리해 장애 격리를 적용하였다.

2. 확장성 측면
- CPU 50% 기준 HPA를 적용해 과제 요구사항을 직접 반영하였다.
- GKE Node Pool autoscaling으로 Pod 확장뿐 아니라 노드 확장도 가능하게 하였다.
- 부하가 집중되는 서비스만 선택적으로 scale-out할 수 있게 분리하였다.

3. 운영성 측면
- Terraform과 Kubernetes 매니페스트로 인프라 및 운영 환경을 코드화하였다.
- Jenkins와 Argo CD를 유지하여 과제 제약사항을 만족하였다.

### 6.2 한계 및 개선 사항

현재 설계는 과제 요구사항을 만족하는 방향으로 적절하지만, 실제 운영 안정성을 더 높이기 위해 다음 보완이 필요하다.

1. readinessProbe / livenessProbe 추가
- 현재 서비스 매니페스트에서 애플리케이션 레벨 readiness/liveness 설정이 확인되지 않는다.
- 무중단 배포와 장애 감지 정확도를 높이기 위해 반드시 추가하는 것이 바람직하다.

2. PodDisruptionBudget 적용
- 노드 유지보수 또는 자발적 축출 상황에서도 최소 가용 Pod 수를 보장할 필요가 있다.

3. topology spread 또는 anti-affinity 적용
- 현재 매니페스트만으로는 동일 서비스 Pod가 서로 다른 노드에 분산 배치된다는 보장이 약하다.
- 단일 노드 장애 대응을 더 확실히 하려면 분산 배치 정책이 필요하다.

4. Argo CD Application 리소스 명시화
- 본 저장소에는 Argo CD 설치 코드는 포함되어 있으나, 운영 애플리케이션 동기화 대상 Application 리소스는 별도 명시가 보이지 않는다.
- GitOps 완결성을 높이려면 Application 또는 ApplicationSet 정의도 함께 저장소에 포함하는 것이 좋다.

5. 정량 측정 자동화
- 부하 테스트 도구와 장애 주입 스크립트를 저장소에 포함하면 Measure 단계의 재현성과 객관성이 높아진다.

### 6.3 종합 평가

정리하면, 본 아키텍처는 Kubernetes 기반의 표준 가용성/확장성 패턴을 활용하여 과제 요구사항을 충족하도록 설계되었다. 특히 핵심 기능 다중화, 데이터 계층 HA, 기능 단위 분리, HPA 및 GKE autoscaling 적용은 요구사항과 직접 연결된다. 향후에는 운영 품질을 더 높이기 위해 무중단 배포 보장 장치와 GitOps 명세 완성도를 보강하는 것이 적절하다.
